# Weekend 4 — Journal

> Running log for the Weekend 4 build (atmosphere: typewriter, grain, modal animation, ambient audio, text jitter). Filled in across the weekend; not a retrospective written after the fact.

## What I built

- Task 1 — Weekend 4 spec, ADRs, and journal stub. Spec lives at `docs/specs/weekend-4-atmosphere.md` (the contract for the five-feature atmosphere pass: typewriter, grain, modal animation, ambient audio, text jitter). Promoted spec §13's two recorded decisions to full ADRs: ADR-0016 (atmosphere + Motion library admission, narrowly scoped to the accusation modal) and ADR-0017 (text jitter via per-suspect `nervousnessTriggers`, frontend-only, with `prefers-reduced-motion` + explicit toggle). ADR-0011 status header rewritten in place to record the partial supersession — the CSS-only clause is the only thing relaxed; noir palette / typography / dark-only stay binding. Journal stub created with the five-section template. No code or runtime behaviour touched; `pnpm typecheck && pnpm lint && pnpm test && pnpm test:e2e` all green.
- Task 2 — Storage migration + state extensions. `Suspect` gains optional `nervousnessTriggers` (`{ keywords; increment }`); `Message` gains uniform `displayedContent: string` (mirrors `content` for user messages; starts at `''` for assistant messages and grows via `tick`); `CaseProgress` gains `nervousnessBySuspect: Record<string, number>`. Storage key bumps to `the-ai-interrogation:game:v2` with `persist` version `1 → 2` and a clean-slate `migrate` callback (any `:v1` data is dropped on first hydration, per ADR-0016). Five new store actions land: `tick`, `skipTypewriter`, `bumpNervousness`, `decayNervousness`, and `getNervousness` selector. `appendUserMessage` is wired so that when the active suspect has `nervousnessTriggers` it bumps on any keyword match and decays otherwise — frontend-only, no API/model changes. Pure helpers extracted to `src/lib/game/nervousness.ts` (`applyNervousnessBump`, `applyNervousnessDecay`, `countNervousnessMatches`) so the keyword-scan and decay logic can be unit-tested without booting the store and without prematurely committing keywords to actual suspects (Task 5's job). 26 new unit tests added (`tests/unit/nervousness.test.ts`, `tests/unit/typewriter.test.ts`); all 71 unit tests + the E2E happy-path stay green. No UI rendering touched — typewriter render and jitter visuals are Task 3 / Task 5.
- Task 3 — Char-by-char typewriter rendering. Extracted per-message rendering into `src/features/interrogation/chat-message/` (ADR-0006 folder + barrel) so a per-message `useTypewriter` hook can mount under a stable component identity. New `src/features/interrogation/use-typewriter.ts` runs a `requestAnimationFrame` loop that accumulates wall-clock time and calls `tick` once per ~45ms (Disco Elysium pacing per spec §3.1) — multiple ticks fire in a single frame if the loop fell behind. Loop self-stops when `displayedContent` has caught up AND streaming has finished. Refresh-snap implemented per spec §3.3: on hook mount, if streaming is already finished but `displayedContent < content`, we call `skipTypewriter` immediately rather than replay or leave the message truncated. Skip button is a small subtle font-sans button next to the bubble; visible only while `displayedContent.length < content.length`. `chat-view` rewritten to map messages into `<ChatMessage>` instances keyed on `message.id`. All four checks green; Playwright happy-path took 3.9s (was 1.1s) because the typewriter delays the assertion text, comfortably under the existing 5s timeout. Manual QA against the real Anthropic API confirmed all five §3.3 cases: char-by-char visible at the right pace, Skip flushes instantly, user messages render directly, refresh mid-typewriter snaps to full content, new questions get fresh typewriter cycles. Console clean, no errors.
- Task 4 — Grain overlay. New `<GrainOverlay />` component at `src/components/grain-overlay/` (ADR-0006 folder + barrel) — a single decorative `<div aria-hidden="true" />` mounted in `src/app/layout.tsx` just inside `<body>` before `{children}`, so every screen (Briefing, Investigation, Outcome) inherits the layer with no per-screen wiring. Procedural SVG noise via inline `<feTurbulence>` data URL (no separate asset, no network request); `mix-blend-mode: overlay` blends the grain into the surface beneath rather than sitting on top; `opacity: 0.06` (low end of spec §4.1's 5–8% range); `pointer-events: none` so the layer never intercepts clicks. CSS lives next to the existing dot-pulse keyframes in `globals.css`. All automatic checks green; manual QA approved by maintainer: subtle texture on all three screens, text fully readable, clicks pass through cleanly. Task 8 sensory QA can dial opacity up if it reads as too subtle in the final mix.
- Task 5 — Text jitter mechanic, the W4 signature feature. Marcus / Henry / Diana got `nervousnessTriggers` verbatim per spec §7.2 (15 / 25 / 10 increments; keyword lists exactly as specified). New presentational wrapper `JitteringText` at `src/components/jittering-text/` (ADR-0006 folder + barrel) picks one of `.jitter-low` / `.jitter-mid` / `.jitter-high` based on §7.4 bands; below 20 it returns children unwrapped so unaffected text has zero overhead. CSS bands plus the `prefers-reduced-motion: reduce` override land in `globals.css` next to grain and dot-pulse keyframes; the `--jitter-intensity` CSS custom property is the only thing the band classes vary, so the keyframe is shared. `ChatView` now computes `lastAssistantIndex` and threads `isLatestAssistant` into `ChatMessage`; only the latest assistant bubble wraps in `<JitteringText>`. Maintainer dialled the typewriter speed down twice in the same task (45 → 25 → 18ms) — 45ms felt sluggish on Henry's long replies, 18ms (~55 chars/sec) reads as a fast typewriter without losing the per-char reveal. All four automatic checks green; manual QA approved by maintainer (typewriter speed feels right, jitter only on the freshest reply at all three bands, Marcus / Henry / Diana intensities feel calibrated, `prefers-reduced-motion` disables jitter, no console errors).
- Task 6 — Accusation modal scale-in animation. First use of `motion` (v12) — the dependency was already in `package.json` from a prior install, but this is the first import; per ADR-0016 it stays scoped narrowly to this modal. Replaced the shadcn `DialogContent` wrapper with Radix primitives directly (`DialogPrimitive.Root` / `.Portal` / `.Overlay` / `.Content`) using `asChild forceMount` so a `motion.div` becomes the actual Radix surface, with `<AnimatePresence>` driving enter/exit. Backdrop animates `opacity 0→1 + backdropFilter blur(0px)→blur(4px)` over 300ms; modal animates `scale 0.85→1 + opacity 0→1` over 400ms with `cubic-bezier(0.16, 1, 0.3, 1)`; exit reverses with a lighter `scale 0.95` so the modal "releases" rather than retracts. The X close button shadcn's wrapper provided is gone — close happens through Cancel / Escape / backdrop click; Cancel is the primary affordance and the modal is small enough that a second close target felt redundant. All four automatic checks green; manual QA approved by maintainer.
- Task 7 — Ambient audio + Atmosphere/Reduce Motion toggles. Single `<AudioController>` mounted in `game-root` drives a rain-only loop on Investigation (volume 0.4 per §6.4), fades out over 1s on Outcome, and stays silent on Briefing — the gesture-grant from the toggle click stays alive across screen changes because the `<audio>` element never unmounts. Toggles persist via a custom `usePersistentToggle` hook over `useSyncExternalStore` (SSR-safe, no React Context plumbing, same-tab + cross-tab updates fan out via an in-process subscriber bus). Dedicated localStorage keys per spec §6.2 / §7.6 — NOT in the game store. **Reload autoplay fallback:** when `play()` rejects after a page reload (browser autoplay policy drops gesture authority across reloads), the controller registers one-shot `click` / `keydown` / `pointerdown` / `touchstart` listeners; the next user gesture transparently resumes the loop. Reduce Motion adds `.reduce-motion` to `<html>`; CSS override uses `!important` per spec §7.6 because plain specificity wasn't enough to beat the `.jitter-*` shorthand under Tailwind v4. Modal scale-in stays on under Reduce Motion (per ADR-0017 design choice — punctuation animation, not ambient). All automatic checks green; manual QA approved by maintainer.

## Spec deviations

- **Task 5 — Typewriter speed: 45ms → 18ms.** §3.1 originally specified 45ms per character (Disco Elysium pacing). During Task 5 manual QA the maintainer dialled this down twice — first to 25ms, then to 18ms (~55 chars/sec) — because 45ms made long Henry replies feel sluggish and broke the conversational pace the rest of the atmosphere features were building. 18ms still reads as a left-to-right char-by-char reveal rather than an instant dump. `TYPEWRITER_INTERVAL_MS` in `src/features/interrogation/use-typewriter.ts` is the single source of truth; §3.1 + §3.2 + §12 row 5 record the deviation. No structural change to the RAF loop or the buffer/queue architecture.
- **Task 5 — Jitter scope: every assistant message → latest only.** §7.5 originally instructed wrapping every assistant message bubble in `<JitteringText>` keyed on the current nervousness. During manual QA this made past replies retroactively start shaking when the player surfaced a new trigger word, which broke the "current reaction" reading the mechanic is meant to deliver. Narrowed scope to the latest assistant message only: `ChatView` computes `lastAssistantIndex` and passes `isLatestAssistant` to `ChatMessage`; only the latest one wraps in `<JitteringText>`, the rest render plain. §7.5 + §12 row 5 record the deviation. Side benefit: zero animation cost on older messages, which compounds with the no-class-below-20 optimisation.
- **Task 7 — Audio loops: rain + room tone → rain only.** §6.1 originally specified two simultaneous loops (gentle rain + quiet room-tone hum), mixed in the browser via two `<audio>` elements. Maintainer descoped the room-tone half during the §10 pre-task asset hunt — rain alone reads as "atmosphere" without the extra body. `AudioController` is a single-element controller; adding a second loop later is a small additive change (another `ref` + another `play`/`pause` branch). §6.1 + §12 row 7 record the deviation. Also recorded inline in §6.1: the shipped rain MP3 is larger than the ≤200KB suggestion in the original brief — accepted since rain is the only loop and the size still fits within the §6.1 "≤300KB total" budget.
- **Task 7 — Reduce Motion CSS `!important`.** §7.6's CSS override (`.reduce-motion .jitter-*`) only consistently beat the base `.jitter-*` `animation` shorthand when marked `!important`. Spec §7.6 already suggested the `!important` form literally; Task 7 manual QA confirmed plain specificity wasn't enough under Tailwind v4's layering. The same `!important` was applied to the `prefers-reduced-motion` media-query override for parity. Not a behavioural deviation — just a CSS-mechanism note worth recording so a future maintainer doesn't strip the `!important` thinking it's a smell.
